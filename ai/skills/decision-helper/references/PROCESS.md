# PrOACT-URL Decision Process Structure

## Framework Overview

Based on *Smart Choices* by Hammond, Keeney, and Raiffa. The acronym stands for:

| Element | Purpose | Can short-circuit after? |
|---------|---------|--------------------------|
| **Pr**oblem | Define what you're actually deciding | Yes - problem may dissolve |
| **O**bjectives | Clarify what you want | Yes - choice may become obvious |
| **A**lternatives | Generate options | Yes - one dominant alternative |
| **C**onsequences | How alternatives meet objectives | Yes - clear winner emerges |
| **T**radeoffs | Resolve competing objectives | Yes - swaps clarify winner |
| **U**ncertainty | What you don't know | Yes - risk profile settles it |
| **R**isk Tolerance | How much risk you can bear | Yes - acceptable risk found |
| **L**inked Decisions | Future decisions this affects | Terminal |

The process is iterative and refinement-based. At any point, if the **user** says the decision
is clear, the process terminates and moves to the Decision Record.

## Facilitator Role Reminder

In every phase below, **all substantive content must come from the user**. The questions listed
are prompts for the facilitator to ask. The facilitator:
- Asks the questions and records the user's answers
- Gives feedback on clarity, completeness, consistency, and specificity of the user's responses
- May suggest that an area feels incomplete and ask if the user wants to explore further
- Must NOT fill in answers, generate lists, evaluate alternatives, or express opinions on content

## Process Phases

### Phase 0: Intake

Capture the raw decision need before any analysis. **Ask the user each of these questions.**

- **Initial decision statement** — Ask: "In your own words, what are you trying to decide?"
- **Trigger** — Ask: "What prompted this decision now?"
- **Urgency / time horizon** — Ask: "When does this need to be decided by?"
- **Stakes assessment** — Ask: "How high-stakes does this feel to you? (low / medium / high / critical)"
- **Decision type tag** — Ask: "What category does this fall into? (career, financial, technical, relational, health, strategic, operational, other)"

Stakes are recorded for context but do not change the process — all phases are covered regardless.

---

### Phase 1: Problem Definition (Pr)

Goal: Ensure the user is solving the right problem with the right scope. **Ask these questions
one or two at a time. Record the user's answers.**

1. **Gains** — Ask: "What can you gain from this situation?"
2. **Trigger analysis** — Ask: "How does the trigger connect to the decision? Why is this coming up now specifically?"
3. **Constraints** — Ask: "What constraints are you working within?" Then for each: "Is that constraint definitely real, or is it an assumption?"
4. **Essential elements** — Ask: "What are the non-negotiable elements of this decision?"
5. **Linked decisions** — Ask: "Are there other decisions that depend on this one, or that this one depends on?"
6. **Scope** — Ask: "What's in scope and what's out of scope for this decision?"
7. **External input** — Ask: "Who else could help clarify this? Have you talked to them?"
8. **Reformulated decision statement** — Ask: "Now that we've explored the problem, how would you restate what you're actually deciding?"

Process feedback: Check that the reformulated statement is specific, bounded, and actionable.
If it's vague, ask the user to sharpen it. Do not rewrite it for them.

**Short-circuit check**: Ask the user — "Has the problem dissolved or become clear?"

---

### Phase 2: Objectives (O)

Goal: Help the user surface what they actually want — both obvious and hidden objectives.
**All objectives must come from the user.** Do not generate objectives yourself.

#### 2a: Explore Concerns (divergent)

Ask these questions to help the user surface latent objectives:

1. **Best case** — Ask: "What would make you really happy about this decision? What's your wishlist?"
2. **Worst case** — Ask: "What do you most want to avoid?"
3. **Others' welfare** — Ask: "How does this affect other people? What do you want for them?"
4. **Precedent** — Ask: "Do you know how others in similar situations have thought about it?"
5. **Ideal fantasy** — Ask: "Imagine an amazing but totally unfeasible outcome. What makes it great?"
6. **Nightmare scenario** — Ask: "Imagine the worst outcome. What makes it terrible?"
7. **Justification** — Ask: "How would you explain or justify this decision to someone you respect?"

#### 2b: Distill Objectives (convergent)

Help the user convert their raw concerns into structured objectives:

1. **Draft objectives list** — Ask: "Based on everything you just said, can you list your objectives as short verb + object phrases? (e.g., 'save money', 'reduce commute', 'preserve autonomy')"
2. **Five Whys per objective** — For each objective the user lists, ask "Why is that important to you?" repeatedly to help them reach the fundamental level
3. **Fundamental objectives list** — Ask: "Which of these are the deep, fundamental things you want — the ends, not the means?"
4. **Means objectives list** — Ask: "Which are means to those ends? (We'll keep these — they're useful for generating alternatives later)"

Process feedback:
- If the list seems short, note: "You have N objectives — do you feel that captures everything, or does something feel missing?"
- If an objective is vague, ask the user to make it more specific
- Do not add, remove, or reorder objectives without the user's direction

**Short-circuit check**: Ask the user — "Does one path obviously satisfy all your fundamental objectives?"

---

### Phase 3: Alternatives (A)

Goal: Help the user generate a rich, varied set of alternatives. **All alternatives must come
from the user.** Do not generate alternatives yourself. Your job is to prompt the user using
different generation methods until they have a varied set.

#### 3a: Self-Generated Alternatives

Ask the user to generate alternatives using these lenses (one or two at a time):

1. **Objective-driven** — Ask: "Looking at each of your fundamental objectives, what alternatives come to mind for achieving them?"
2. **Constraint-challenging** — Ask: "What alternatives would exist if you challenged one of your constraints?"
3. **Constraint-free** — Ask: "If you could remove one key constraint entirely, what would you do?"
4. **Wildly ambitious** — Ask: "What would you do with unlimited resources?"

#### 3b: External Alternatives

1. **Research** — Ask: "Have you looked into how others have handled similar situations? What did you find?"
2. **Consultation** — Ask: "Have you asked anyone else for suggestions? What did they come up with?"
3. **Incubation** — Ask: "Have you given yourself time to just sit with this? Sometimes alternatives surface after stepping away."

#### 3c: Process & Special Alternatives

1. **Process alternatives** — Ask: "Would a decision process itself work here? Like delegating it, flipping a coin, or scoring?"
2. **Win-win alternatives** — Ask: "Is there a way to reframe this so multiple parties get what they want?"
3. **Information-gathering alternatives** — Ask: "Would more research help? Is there info you could gather that would change your options?"
4. **Time-buying alternatives** — Ask: "Is there a way to buy more time before committing?"

#### 3d: Alternative Sufficiency Check

Ask the user to confirm:
- [ ] "Have you explored all the generation methods above?"
- [ ] "Would you be satisfied choosing one of these alternatives right now?"
- [ ] "Do you have genuine variety in your alternatives?"
- [ ] "Do you want to add anything else before we move to evaluating consequences?"

Process feedback: If the user has fewer than 3 alternatives, or they all look very similar,
note this: "You have N alternatives and they seem quite similar. The framework suggests pushing
for more variety. Want to try another generation method?" Do not suggest specific alternatives.

**Short-circuit check**: Ask the user — "Is one alternative clearly dominant across all your objectives?"

---

### Phase 4: Consequences (C)

Goal: Help the user understand how each alternative performs against each objective.
**All evaluations must come from the user.** You structure the conversation; they provide
the assessments.

#### 4a: Free-Form Descriptions

For each alternative, ask the user:
1. "Walk me through what would happen if you chose [alternative]. What are the consequences?"
2. "Can you be specific — numbers, timelines, concrete details?"
3. After all alternatives are described: "Did describing these consequences surface any new objectives you hadn't thought of?"

Process feedback: If descriptions vary wildly in detail (one paragraph vs. one sentence),
note: "Your description of Alternative 2 is much less detailed than Alternative 1. Want to
flesh it out so they're comparable?"

#### 4b: King of the Hill Elimination

1. Ask: "Which alternative looks strongest to you right now?"
2. For each other alternative, ask: "How does [alternative] compare to your strongest? Is it clearly worse, or does it compete?"
3. Ask: "Which alternatives do you want to cut, and why?"

#### 4c: Consequences Table

Help the user build a structured table:
- Ask: "For each of your objectives, how does each remaining alternative perform? Let's go objective by objective."
- Walk through each objective, asking the user to rate or describe each alternative's performance
- The user provides all cell values; you structure them into the table

Process feedback — check with the user:
- "Are you using consistent scales across alternatives for each objective?"
- "Are there objectives where you're relying on gut feel? That's fine, but let's name it."
- "Is there data you could look up that would make any of these more precise?"

#### 4d: Pairwise Comparison

Ask: "Let's compare the remaining alternatives two at a time. For [A] vs [B], what are the pros of A over B? And the cons?"

Ask: "Based on these comparisons, do you want to eliminate any?"

**Short-circuit check**: Ask the user — "Does one alternative clearly dominate?"

---

### Phase 5: Tradeoffs (T)

Goal: Help the user resolve competing objectives when no alternative wins on everything.
**The user makes all tradeoff judgments.** You walk them through the method.

#### 5a: Dominance Check

1. Ask: "Looking at your consequences table, can you rank the alternatives for each objective? (1st, 2nd, 3rd...)"
2. Ask: "Are any alternatives worse on every single objective? Those can be eliminated."
3. Ask: "Are any alternatives worse on most objectives and only barely better on the rest? Worth cutting?"

#### 5b: Even Swaps

Walk the user through the even swap method:

1. Ask: "Which objective would be easiest to make equal across your remaining alternatives?"
2. For each pair, ask: "What would need to change to make [Alt A] and [Alt B] equal on [objective]? And what change in another objective would you consider a fair trade for that?"
3. Ask: "Now that [objective] is equalized, we can set it aside. Do any alternatives now look dominated?"
4. Repeat until the user has a clear preference or a small final set.

Process feedback:
- If the user struggles with a swap, suggest trying an easier objective first
- If swaps seem inconsistent, note: "Earlier you valued X at [amount] but this swap implies a different value. Want to revisit?"

**Short-circuit check**: Ask the user — "Is the decision clear? If yes, let's capture your Decision Record."

---

### Phase 6: Uncertainty (U)

Goal: Help the user account for what they don't know. **The user identifies uncertainties
and assesses likelihoods.** You ask the questions.

1. **Key uncertainties** — Ask: "What important things are unknown or unpredictable that could affect your decision?"
2. **Scenarios** — For each uncertainty, ask: "What are the plausible outcomes for [uncertainty]?"
3. **Likelihood assessment** — Ask: "How likely is each scenario? (very unlikely / unlikely / possible / likely / very likely)"
4. **Impact on consequences** — Ask: "How would [scenario] change the consequences for your alternatives?"
5. **Risk profiles** — Ask: "For each remaining alternative, what's the range of outcomes — best case, most likely, worst case?"
6. **Dominance under uncertainty** — Ask: "Does one alternative hold up well across most or all of these scenarios?"

Process feedback: If the user lists only one or two uncertainties, note: "You've identified N
uncertainties. Are there others you might be overlooking — things that are easy to assume will
stay constant but might not?"

**Short-circuit check**: Ask the user — "Does one alternative perform acceptably across all plausible scenarios?"

---

### Phase 7: Risk Tolerance (R)

Goal: Help the user match the decision to their appetite for risk. **The user assesses their
own risk tolerance.** You ask the questions.

1. **Risk stance** — Ask: "For this specific decision, are you risk-averse, risk-neutral, or risk-seeking? Why?"
2. **Desirability assessment** — Ask: "How much do you value the best possible outcome vs. how much do you fear the worst? Where's the balance for you?"
3. **Risk mitigation** — Ask: "Can you reduce the downside of your preferred alternative? How?"
4. **Diversification** — Ask: "Is there a way to hedge — split across alternatives or keep options open?"
5. **Insurance** — Ask: "Can you create a fallback or safety net?"

Process feedback: If the user's stated risk stance seems inconsistent with their earlier
tradeoff choices, note: "You said you're risk-averse, but in the tradeoffs phase you chose
the higher-variance option. Can you reconcile those?"

**Short-circuit check**: Ask the user — "Is the risk-adjusted choice clear?"

---

### Phase 8: Linked Decisions (L)

Goal: Help the user consider how this decision connects to future decisions. **The user
identifies the links.** You ask the questions.

1. **Future decisions enabled/foreclosed** — Ask: "What doors does each alternative open or close for you?"
2. **Information value** — Ask: "Will this decision reveal information that affects future choices?"
3. **Reversibility** — Ask: "How reversible is each alternative? What would it cost to change course?"
4. **Sequencing** — Ask: "Should any part of this decision be deferred? Does the order matter?"
5. **Decision tree** — If the linked decisions are complex, ask: "Want to sketch out how these decisions chain together?"

Ask: "Considering all of this — linked decisions, reversibility, sequencing — does your preference change?"

---

## Decision Record

The terminal output of the process. Every decision produces this, regardless of how deep the process went.
